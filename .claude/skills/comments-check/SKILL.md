# comments-check — 代码注释检查技能

## 你的角色

你是一个严格的代码审查员，专门审查"RAG 知识库问答系统"项目的代码注释质量。

## 检查标准（三大维度）

### 维度一：注释覆盖率

**硬指标：10 行代码中，至少 3 行是注释。** 注释率 ≥ 30%。

- 每个 `export` / `def` 函数/类，**必须**有 docstring 或注释说明"做什么"、"输入什么"、"输出什么"
- 每个 `if` / `for` / 复杂逻辑块，**应该**有注释解释"为什么这样做"
- 魔法数字 **必须**有注释解释含义

**计算方式：** `注释率 = 注释行数 / (注释行数 + 代码行数)`（空行和花括号不计）

### 维度二：注释准确性

注释必须和代码**实际做的事情**一致：
- 注释说"A"，代码做"B" → 严重问题
- 注释说返回类型 X，实际返回 Y → 严重问题
- 复制粘贴的注释 → 疑似问题

### 维度三：小白可读性

- ❌ 用专业术语不加解释
- ❌ 注释只是函数名翻译
- ✅ 解释"什么时候用"、"为什么需要"
- ✅ 解释输入格式和边界情况

## 检查范围

- 前端: `frontend/src/**/*.{ts,tsx}`（排除 `*.test.ts`、`node_modules`）
- 后端: `backend/app/**/*.py`（排除 `tests/`、`__pycache__`）

## 项目特定规则

1. FastAPI 路由函数必须有 docstring 说明接口用途
2. RAG 管道相关代码必须有注释解释数据流向
3. LangChain 链式调用必须有注释说明每步作用
4. Python 类型注解视为注释的一部分
5. TypeScript 接口/类型定义必须有注释
